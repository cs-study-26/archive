## ✅ 실행 계획(Execution Plan)이란?

> [!NOTE]
> **실행 계획**은 사용자가 SQL을 던졌을 때, 옵티마이저가 수립한 **'최종적인 처리 경로'**를 말합니다. 우리가 내비게이션에 목적지를 찍었을 때 나오는 '최적 경로'와 같습니다. 어떤 인덱스를 쓸지, 어떤 순서로 테이블을 조인할지 등이 이 계획에 담깁니다.



<br>

## ✅ EXPLAIN 명령어의 활용

실행 계획을 확인하려면 쿼리 앞에 [EXPLAIN PLAN] 키워드를 붙여 실행합니다.
```sql
EXPLAIN PLAN
SELECT * FROM 테이블명 WHERE 조건
```
EXPLAIN PLAN은 SQL의 실행 계획만을 확인할 수 있고 명령을 사용할 때 데이터를 처리하지 않습니다. 그렇기 때문에 데이터베이스에 그 어떤 부하도 걸리지는 않습니다.
다만 EXPLAIN PLAN은 한번에 하나의 쿼리문의 실행계획만을 확인할 수 있고 그 실행계획을 또 확인하기 위해 별도의 SELECT문을 작성해야 하는 번거로움과 데이터를 읽지 않기 때문에 소요시간을 추정한다거나 데이터 관련 I/O정보를 확인할 수 없다는 점이 단점입니다.

<br>

## ✅ Plan 요소
### 💡 1. cost (비용)
- Cost는 누적된 값이며, 하위(Child)의 Cost를 RollUp 합니다. -> 마지막에 읽는 Cost가 모든 쿼리문에 대한 누적 Cost
- 옵티마이저가 측정한 오퍼레이션 수행에 필요한 예측 비용입니다.
- 쿼리를 수행함에 있어 사용된 자원이나, 작업의 단위입니다.
- 대체로 적을 수록 효율적인 쿼리(좋은 성능)입니다.
- Full Scan을 해야 하는 쿼리는 의미가 없을 수도 있습니다.
- cost 수치가 더 높은 질의문이 빠른 경우도 있습니다.

### 💡 2. Cardinality
- 행 집합에서 행의 수를 표시합니다.(분포도)
- 행 집합은 기본 테이블, 뷰, 조인이나, GROUP BY의 결과입니다.
- 행 집합을 의미하니 적게 나타날수록 SQL이 빠를 수 있습니다.
- 옵티마이저가 측정합니다.

### 💡 3. Bytes
- 각 실행계획 단계에서 Access된 byte 수를 의미합니다.
- 옵티마이저가 측정합니다.

<br>

## ✅ 실행 계획을 통해 알 수 있는 것들

1. **인덱스 전략의 유효성:** 내가 만든 인덱스를 DB가 실제로 쓰고 있는지, 아니면 무시하고 풀 스캔을 하는지 알 수 있습니다.
2. **조인 순서:** 여러 테이블을 조인할 때 어떤 테이블을 먼저 읽는지(Driving Table) 확인하여 조인 효율을 판단합니다.
3. **병목 구간:** 어떤 단계에서 가장 많은 행(rows)을 검사하는지 찾아내어 튜닝 포인트를 짚어낼 수 있습니다.
4. **정렬 및 그룹화 오버헤드:** 인덱스를 활용해 정렬을 처리하는지, 아니면 별도의 물리적 정렬 작업을 하는지 알 수 있습니다.

## ✅ 실행 계획(Execution Plan) 분석과 확인

---

> [!IMPORTANT]
> **실행 계획(Plan)**은 옵티마이저가 수립한 최종 결정문입니다. 이를 분석하는 것이 쿼리 튜닝의 첫걸음입니다.

### 💡 실행 계획 확인 방법
```sql
EXPLAIN SELECT * FROM members WHERE email = 'test@gmail.com';
```

### 💡 핵심 항목 분석 가이드 (MySQL 기준)
1. **type (접근 방식):**
   - const/eq_ref: PK나 Unique 키로 딱 한 건을 찾을 때 (최상)
   - ref: 인덱스로 여러 건을 찾을 때 (우수)
   - range: 인덱스 범위 검색 (양호)
   - index: 인덱스 전체 스캔 (비효율)
   - ALL: 테이블 전체 스캔 (최악, 튜닝 대상)
2. **rows:** 쿼리 수행을 위해 옵티마이저가 예상하는 검사 대상 행의 개수입니다. 이 값이 실제 결과값보다 지나치게 크면 인덱스 효율이 낮은 것입니다.
3. **Extra:**
   - Using index: **커버링 인덱스** 적용 (성능 매우 좋음)
   - Using filesort: 정렬용 인덱스가 없어 별도 정렬 수행 (성능 저하 원인)
   - Using temporary: 임시 테이블 생성 (메모리/디스크 부하 발생)
